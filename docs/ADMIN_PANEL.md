# Панель администратора (Admin Panel) и Управление БД

Веб-панель администратора предоставляет интерфейс управления пользователями, просмотра логов аудита, а также редактирования контактов телефонного справочника. 

Панель построена на FastAPI (backend) и Vanilla JS/HTML/CSS (SPA frontend). По умолчанию запускается на порту `8080` при использовании скрипта `run_all_bots.py` или `run_server.py`.

---

## Возможности панели

1. **Управление администраторами**: Создание, изменение ролей (Супер-администратор, Администратор) и распределение прав (`edit_contacts`, `view_audit_logs`).
2. **Управление телефонным справочником**: 
   - Полноценный CRUD (создание, чтение, обновление, удаление) контактов в базе SQLite (`data/contacts.db`).
   - Регистронезависимый поиск на русском языке для любых полей (ФИО, телефон, email, предприятие, отдел).
   - Автоматическая алфавитная сортировка по умолчанию (`company ASC, department ASC, full_name ASC`).
3. **Логирование аудита**: Запись действий администраторов (кто, когда и что изменил, с детализацией старых и новых значений полей) в базу данных PostgreSQL (таблица `admin_audit_logs`).
4. **Синхронизация с поисковым индексом (Qdrant)**:
   - При внесении изменений в контакты устанавливается флаг о необходимости синхронизации.
   - Фоновая задача каждую ночь в `03:00` проверяет этот флаг и запускает переиндексацию контактов в Qdrant.
   - Супер-администратор может запустить синхронизацию вручную в любой момент из панели.

## Архитектура базы данных SQLite и WAL-режим

Справочник контактов хранится в локальном файле `data/contacts.db` (SQLite). Для обеспечения надежной работы и исключения ошибок взаимных блокировок (например, `database is locked`) при одновременной работе нескольких администраторов или ботов, применены следующие архитектурные решения:

1. **WAL-режим (Write-Ahead Logging)**:
   - При подключении к базе данных выполняются команды:
     ```sql
     PRAGMA journal_mode=WAL;
     PRAGMA synchronous=NORMAL;
     ```
   - В WAL-режиме операции чтения (поиск по справочнику ботами) не блокируют операции записи (редактирование контактов администраторами) и наоборот.
   - **Особенность на Ubuntu**: В WAL-режиме SQLite создает временные файлы `contacts.db-wal` и `contacts.db-shm` в той же директории, где лежит основная БД. Процесс приложения должен иметь права на запись в директорию `data/`.

2. **Таймаут транзакций**:
   - Время ожидания освобождения блокировок базы увеличено до `30.0` секунд для предотвращения сбоев при пиковых нагрузках.

## Автоматическое резервное копирование (Backups)

Перед выполнением любой операции записи (создание, изменение, удаление контакта) бэкенд панели администратора автоматически создает бэкап базы данных:
- **Метод**: Используется безопасное API онлайн-копирования `sqlite3.Connection.backup()`. Оно корректно учитывает состояние WAL-журналов и создает копию «на лету» без блокировки чтений.
- **Производительность**: Операция выполняется в фоновом пуле потоков через `asyncio.to_thread` и не блокирует основной цикл событий (event loop) FastAPI.
- **Хранение**: Бэкапы сохраняются в директорию `data/backups/` в формате `contacts_backup_YYYYMMDD_HHMMSS.db`.
- **Ротация**: Настроена автоматическая ротация резервных копий. В системе хранится не более **30** последних копий. Настройка количества хранимых копий находится в `admin/app.py` в константе `BACKUP_RETENTION_COUNT`.

## Развертывание: Разработка (Windows) vs Продакшн (Ubuntu)

Проект разрабатывается локально на Windows, но работает на сервере под управлением **Ubuntu** через Docker Compose. 

### Особенности для продакшена на Ubuntu:

1. **Сохранение данных (Persistent Volumes)**:
   Поскольку PostgreSQL и само FastAPI-приложение запускаются в контейнерах, база данных SQLite (`contacts.db`) и папка с бэкапами (`backups/`) должны сохраняться при пересоздании контейнеров или при выполнении команды `docker compose down -v`.
   Для этого в `docker-compose.yaml` настроен проброс (bind mount) директории `data` с хост-системы:
   ```yaml
   services:
     app:
       ...
       volumes:
         - ./data:/app/data
   ```
   *Все бэкапы и файлы базы данных гарантированно сохраняются на жестком диске сервера Ubuntu в папке `./data/`.*

2. **Права доступа (File Permissions) на Ubuntu**:
   - При работе в Docker или при запуске в качестве systemd-службы на Linux, пользователь, от имени которого работает процесс, должен иметь полные права на чтение и запись в папку `./data/`.
   - Если прав на запись в директорию недостаточно, SQLite не сможет создать вспомогательные файлы разделяемой памяти и журнала (`.db-shm`, `.db-wal`), и приложение выдаст ошибку `ReadOnlyError` или `Permission Denied`.
   - Рекомендуется настроить права на сервере:
     ```bash
     chmod -R 775 ./data
     chown -R 1000:1000 ./data  # (или ID пользователя внутри контейнера)
     ```

3. **Копирование бэкапов**:
   При необходимости ручного копирования базы данных `contacts.db` с сервера Ubuntu (например, для локальной отладки), не копируйте файл напрямую при запущенном и активно пишущем приложении. Лучше скопируйте самый свежий файл бэкапа из директории `data/backups/`, так как он гарантированно целостен.
