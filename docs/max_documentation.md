# Полная техническая документация платформы MAX: Чат-боты и API

## 1. Подготовка и настройка бота
* [cite_start]**Уровень сложности**: Продвинутый[cite: 3].
* [cite_start]**Доступность**: Подключение к платформе MAX для партнёров доступно для юрлиц и ИП, являющихся резидентами РФ[cite: 2].
* [cite_start]**Возможности**: С навыками разработки можно создавать ботов с неограниченным потенциалом и размещать мини-приложения в MAX[cite: 4].
* [cite_start]**Требование к профилю**: Создать бота можно только при наличии верифицированного профиля организации на платформе MAX для партнёров[cite: 6].
* [cite_start]**Лимиты**: Для одной организации доступно создание 5 ботов[cite: 7].
* **Модерация**: Доступ пользователей открывается после успешной модерации; [cite_start]статус отображается рядом с названием бота[cite: 8].

## 3. Токен и API-взаимодействие
### Токен бота
* [cite_start]**Получение**: Появляется после проверки в разделе *Чат-боты → Интеграция → Получить токен*[cite: 21].
* [cite_start]**Обновление**: Токен можно обновить в панели управления[cite: 23, 130].
* **Безопасность**: Токен — прямой доступ к боту. [cite_start]Не храните в открытых источниках[cite: 25].

### Работа с запросами
* [cite_start]**Принцип**: Бот отправляет запросы к API MAX и получает JSON в ответ[cite: 30].
* **Авторизация**: Передача через query-параметры не поддерживается. [cite_start]Используйте заголовок `Authorization: <token>`[cite: 31, 142].
* [cite_start]**Базовый URL**: `https://platform-api.max.ru`[cite: 38].
* [cite_start]**Ограничение (RPS)**: Максимум 30 запросов в секунду[cite: 38, 183].
* **Методы**: 
    * [cite_start]`GET` (получить), `POST` (создать), `PUT` (редактировать), `DELETE` (удалить), `PATCH` (исправить) [cite: 150-154].

### Коды ответов HTTP
* [cite_start]`200` — Успех[cite: 172].
* [cite_start]`400` — Недействительный запрос[cite: 173].
* [cite_start]`401` — Ошибка аутентификации[cite: 174].
* [cite_start]`404` — Ресурс не найден[cite: 175].
* [cite_start]`429` — Превышен RPS[cite: 177].
* [cite_start]`503` — Сервис недоступен[cite: 178].

## 4. Уведомления (Webhook и Long Polling)
* [cite_start]**Webhook**: Рекомендуется для production-окружения[cite: 46]. [cite_start]Поддерживает только HTTPS (включая самоподписанные сертификаты)[cite: 51, 194].
* [cite_start]**Long Polling**: Только для разработки и тестирования[cite: 46, 181].
* [cite_start]**Настройка**: POST-запрос на `/subscriptions`[cite: 49]. [cite_start]Для получения вебхуков включите тип `bot_started` в параметре `update_types`[cite: 80].

## 5. Диплинки (Deep Links)
* [cite_start]**Формат**: `https://max.ru/<botName>?start=<payload>`[cite: 60].
* [cite_start]**Payload**: До 128 символов[cite: 63].
* [cite_start]**Множественные параметры**: Кодируются в строку: `?start=param1_value1_param2_value2`[cite: 104].
* [cite_start]**JSON ответа (`bot_started`)**: Содержит `update_type`, `timestamp`, `chat_id`, объект `user` и `payload` [cite: 85-94].

## 6. Клавиатура и Форматирование
### Inline-клавиатура
* **Размер**: До 210 кнопок (30 рядов по 7 кнопок)[cite: 190].
* **Типы кнопок**:
    * `callback`: Событие `message_callback`[cite: 193].
    * [cite_start]`link`: Открыть ссылку (до 2048 симв.)[cite: 191, 195].
    * [cite_start]`request_contact` / `request_geo_location`: Запрос контакта или локации[cite: 196, 197].
    * [cite_start]`open_app`: Открыть мини-приложение[cite: 198].
    * [cite_start]`message`: Отправить текстовое сообщение[cite: 199].
    * `clipboard`: Копировать содержимое `payload` в буфер обмена[cite: 200, 202].

### Форматирование текста
Включается через свойство `format`[cite: 237, 240].

| Элемент | Markdown (`format: markdown`) [cite: 238] | HTML (`format: html`) [cite: 241] |
| :--- | :--- | :--- |
| Курсив | `*текст*` или `_текст_` | `<i>` или `<em>` |
| Жирный | `**текст**` или `__текст__` | `<b>` или `<strong>` |
| Зачеркнутый | `~~текст~~` | `<del>` или `<s>` |
| Подчеркнутый | `++текст++` | `<ins>` или `<u>` |
| Моно | \`код\` | `<pre>` или `<code>` |
| Ссылка | `[Текст](url)` | `<a href="url">Текст</a>` |
| Упоминание | `[Имя](max://user/id)` | `<a href="max://user/id">Имя</a>` |

## 9. Особенности разработки и подводные камни (выявлено на практике)
При интеграции бота с платформой MAX (Python, Long Polling / `AIOHTTP`) мы выявили несколько важных неочевидных технических нюансов, которые отличаются от Telegram-бота:

* **Long Polling и `/subscriptions`**: В официальной документации упоминается настройка подписок через метод `/subscriptions`. Однако этот метод предназначен **только для Webhook**. При использовании Long Polling (`GET /updates`), попытка отправить POST-запрос на `/subscriptions` без передачи URL вебхука приведет к ошибке API `400: Field 'url' cannot be null`. Для Long Polling подписки регистрировать не требуется — сервер копит все события автоматически.
* **Фильтрация событий в Long Polling (`types`)**: Хотя документация часто подразумевает возможность фильтрации событий (например, передачу `types=bot_started,message_created` в качестве параметра GET-запроса `/updates`), на практике наличие параметра `types` может полностью заблокировать доставку событий (сервер всегда возвращает пустой массив). Надежнее вызывать `/updates` только с параметрами `timeout` и `marker`, а фильтровать события по `update_type` уже на стороне приложения.
* **Специфичная структура событий (`chat_id`)**: При обработке события `message_created`, идентификатор диалога, в который бот должен отправить ответ (`chat_id`), находится не в корне события `event.chat.chat_id`, а по глубоко вложенному пути: `event.message.recipient.chat_id`.
* **Тег форматирования для MAX**: Для применения HTML-форматирования к тексту, в API запросах необходимо явно передавать `format: "html"` внутри Payload. Иначе теги `<b>` или `<i>` отобразятся пользователю в сыром виде.
* **Работа с вложениями и медиа-файлами**:
    * **Трехэтапная загрузка изображений**: Загрузка картинок (в отличие от обычных файлов) требует строгого соблюдения трех шагов:
        1.  **Запрос ссылки**: POST-запрос на `/uploads?type=image` для получения `upload_url`.
        2.  **Загрузка байтов**: POST-запрос на полученный URL. **Критически важно**: использовать формат `multipart/form-data`, имя поля `v1` и ОБЯЗАТЕЛЬНО указывать параметр `filename="image.png"` (или другое расширение) в заголовке `Content-Disposition`. Без `filename` сервер вернет `400 BAD_REQUEST`. Рекомендуется использовать библиотеку `requests` для этого шага, так как она формирует заголовки корректнее, чем асинхронные аналоги.
        3.  **Отправка сообщения**: Использование токена из ответа Шага 2 в структуре `{"type": "image", "payload": {"token": "..."}}`.
    * **Декодирование токенов**: Токены, возвращаемые медиа-сервером, могут содержать зашифрованные символы. Перед отправкой в `/messages` их необходимо пропустить через `urllib.parse.unquote()`, иначе API вернет ошибку `Invalid photo token`.
    * **Скачивание медиа**: При скачивании файлов/голосовых по URL из объекта `attachments` необходимо использовать "чистую" сессию (например, `aiohttp.ClientSession()`) **без** заголовков авторизации MAX API.
    * **Ошибки CDN (attachment.not.ready / video.not.processed)**: Загрузка сгенерированных голосовых ответов обратно на сервера MAX работает нестабильно. Платформа уводит отправленные файлы на длительный transcoding, из-за чего бот зависает (часто отправляя `attachment.not.ready`), пока файл не обработается.
    * **Рекомендованная архитектура (STT)**: Для приема голосовых сообщений рекомендуется схема **Голос -> Текст (STT)**: скачивание голосового сообщения, распознавание речи (через сторонние LLM) и ответ в текстовом формате. Это исключает задержки и ошибки загрузки вложений.
* **Совместный запуск ботов (MAX + Telegram)**: При использовании многопоточности (запуск Telegram-бота в отдельном `threading.Thread`, а MAX-бота в основном цикле `asyncio`), крайне важно избегать Singleton-паттерна для сетевых клиентов. API-клиенты (`GenAI`, `aiohttp`) привязываются к событийному циклу (Event Loop) того потока, в котором были созданы. Их нельзя передавать между потоками, иначе возникнет фатальная ошибка `got Future attached to a different loop`. Используйте хранилище `threading.local()`, чтобы у каждого потока был свой экземпляр сетевых клиентов.
